# Contributing (GSSOC 2026)

Thanks for your interest in contributing to PESiMENs. This public repository hosts the frontend UI and public docs, and is the community-facing hub for GSSOC 2026.

## Scope of contributions

Public contributions are welcome in this repo for:

- frontend UI components and pages
- landing page improvements
- reusable utilities and hooks in the frontend
- documentation and beginner-friendly tasks

Core backend services and infrastructure remain private.

See [BEGINNER_TASKS.md](BEGINNER_TASKS.md) for starter-friendly ideas.

**We genuinely appreciate every contribution. Contributors will be recognized in the in-app About Us section to honor their work.**

## How to get started

1. Find an issue labeled for GSSOC 2026.
2. Comment on the issue to request assignment.
3. Wait for maintainer confirmation before starting work.
4. Fork the repo, branch from `dev`, and open a PR back into `dev` unless the maintainer says otherwise.

## Making changes

- Keep changes focused to the assigned issue scope.
- Use the frontend lint/format setup before opening a PR.

## Proposing features or improvements

- Open a new issue with a clear title and a short problem statement.
- Include expected behavior, constraints, and any user impact.
- If the idea requires private repo access, mention why and request read access.

## Pull request process (public repo)

- Create a feature branch from `dev` with a descriptive name.
- Keep PRs small and scoped to a single issue.
- Include screenshots or recordings for UI changes.
- Link the issue in the PR description and target `dev` as the base branch.

## Code style requirements

- Follow the existing linting and formatting setup in this repo.
- Keep naming consistent with nearby code.
- Avoid introducing new dependencies without approval.

## Reporting bugs

- Open an issue with steps to reproduce, expected behavior, and actual behavior.
- Include logs or screenshots if safe to share publicly.
- Do not post private code, credentials, or internal links.

## GSSOC 2026 guidelines

- Use the GSSOC-labeled issues as the source of truth.
- One issue per contributor at a time unless approved.
- Maintain respectful communication and follow the project timeline.
- Ask for private access only when necessary.

## Private access policy (backend access)

PESIMENS is being developed as a startup product, so the core production repository remains private. Backend access is granted only on a case-by-case basis by the maintainer. Being part of GSSOC does not automatically grant private access.

### Eligibility criteria

You may be considered for backend access if you meet the following:

- You have a confirmed issue assignment in this public repo.
- You can show relevant experience or past contributions (PESiMENs or similar projects).
- You can explain which private component you need to touch.
- You agree to keep private code and internal details confidential and do not share them in public channels.

### How to request access

1. Comment on your assigned issue with a short access request.
2. Include: your GitHub username, the component you need, and the smallest scope of access required.
3. Wait for maintainer approval. Invites are sent through GitHub and must be accepted.

### Access levels

- **Read**: view-only access to the private repo.
- **Write**: can push branches and open PRs (most common for active contributors).
- **Triage**: manage issues and labels, no code push.

Access can be adjusted or revoked at any time based on activity or policy compliance.

### Important notes

- Do not post private code, screenshots, stack traces, or internal links in public issues or PRs.
- Use private channels for sensitive discussions as directed by the maintainer.
- If you are unsure whether something is private, treat it as private and ask first.

## Security

If you find a security issue, do not open a public issue. Follow the guidance in [SECURITY.md](SECURITY.md).
